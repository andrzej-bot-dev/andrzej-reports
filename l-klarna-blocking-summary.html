🔴 The Problem in 30 Seconds

l.klarna.com is Klarna's branded short link domain used in marketing emails, push notifications, and the "Sign in" button on klarna.com. It doesn't resolve directly — it has a CNAME chain:

CNAME l.klarna.com What users see
CNAME getklarna.customlinks.appsflyer.com 🚫 This is what gets blocked
CNAME onelinkalbnew.onelink.me.cdn.cloudflare.net AppsFlyer OneLink CDN
A 104.18.39.175 / 172.64.148.81 Cloudflare
TL;DR (corrected after multi-round verification): l.klarna.com was directly listed in HaGeZi's raw source file blocklist-referral-native.txt (confirmed via git history, removed July 22). It was simultaneously on the companion allowlist, so compiled HaGeZi output excluded it. The 19,603 AppsFlyer entries are a category error — they're appsflyersdk.com SDK subdomains, not the OneLink/customlinks infrastructure. customlinks.appsflyer.com appears in zero compiled blocklists. Public DoH tests against NextDNS and AdGuard DNS default endpoints show l.klarna.com resolving normally — we could not reproduce blocking. The only primary evidence of actual blocking is the three GitHub issues filed by Klarna engineer v-ilinskiy-klarna. The blocking mechanism may require service-specific configuration we cannot reproduce from outside.

🎯 Revised Block Rate Estimate

After extensive verification — including independent reproduction by a second AI analyst who challenged the initial findings — the estimate has been revised significantly downward. The key discovery was that customlinks.appsflyer.com (the actual CNAME target) is not in any public blocklist, and the default public DoH endpoints of both NextDNS and AdGuard DNS resolve it without blocking.

What we could NOT independently verify: The Klarna engineer reports that NextDNS (with Ads & Trackers) and AdGuard DNS block l.klarna.com. Our DoH tests against the default public endpoints of both services show customlinks.appsflyer.com resolving normally. This may mean: (a) NextDNS requires a specific configuration ID to apply blocklists, (b) AdGuard DNS has server-side CNAME-cloaking rules not present in their public filter.txt, or (c) the blocking is intermittent or app-context-specific. The Klarna engineer is a credible primary source, but we could not reproduce the block from outside.

Why the estimate dropped from 2.6%/10% → 0.2–1%

  1. customlinks.appsflyer.com is in ZERO public blocklists. Not in HaGeZi (any tier), not in StevenBlack, not in AdGuard DNS filter.txt. The 19,603 appsflyer entries are all appsflyersdk.com SDK analytics subdomains — a completely different AppsFlyer subsystem.
  2. Pi-hole doesn't block it — no CNAME following, and l.klarna.com isn't on any list.
  3. Browser extensions don't block it — EasyPrivacy, uBlock Origin, Firefox Disconnect have zero appsflyer entries.
  4. Brave Aggressive is opt-in — and even then, top-level navigation CNAME uncloaking behavior is uncertain.
  5. Default public DoH endpoints resolve it — NextDNS (dns1.nextdns.io) and AdGuard DNS (dns.adguard-dns.com) both return valid IPs for customlinks.appsflyer.com.
  6. HaGeZi Normal (most popular tier) has zero appsflyer entries. Only 64 appsflyer.com subdomains in Ultimate (all SDK/analytics, not OneLink).
What is NOT in dispute: The Klarna engineer reports real user-facing breakage. Three GitHub issues were filed. AdGuard already has a precedent exception for PayMaya (paymaya.customlinks.appsflyer.com). The problem exists — but its blast radius is narrower than both the original report (2.6–10%) claimed, because the blocking mechanism is service-specific server-side CNAME detection, not list-based blocking.

⚔️ Independent Review — Scorecard

An independent analysis challenged several claims in the original report. After honest verification, here's the scorecard:

Claim: "19,603 looks fabricated" WRONG (number is real)

Verified live: curl -sL ".../ultimate.txt" | grep -ci appsflyer → 19,603. The number is real and reproducible. However, the independent reviewer correctly noted that these are almost entirely appsflyersdk.com per-app SDK endpoints — a different AppsFlyer subsystem than the OneLink/customlinks infrastructure that l.klarna.com uses. Real number, but category error when cited as evidence for l.klarna.com blocking.

Claim: "customlinks.appsflyer.com is NOT in any blocklist" CORRECT

Independently verified: customlinks.appsflyer.com appears in zero public blocklists — not HaGeZi (any tier), not StevenBlack, not AdGuard DNS filter.txt, not EasyPrivacy. The original report's implication that the 19,603 entries prove blocking exposure for l.klarna.com was misleading.

Claim: "Default public DoH endpoints resolve it" CORRECT

Both dns1.nextdns.io (NextDNS default) and dns.adguard-dns.com (AdGuard DNS default) resolve l.klarna.com AND customlinks.appsflyer.com without blocking. Full CNAME chain is returned with valid Cloudflare IPs. We could not reproduce any blocking from outside. The Klarna engineer's reported blocking may require service-specific configuration (NextDNS profile with particular blocklists, AdGuard personal config) that the default public endpoints do not apply.

Claim: "Brave CNAME uncloaking is not default" CORRECT

Brave's wiki confirms: CNAME uncloaking is only in Aggressive mode (opt-in). Standard mode is default. The original report's browser-level estimate was overstated.

Claim: "AdGuard 91M users is self-reported marketing" FAIR POINT

AdGuard's user numbers are self-reported and unaudited. Using them as a load-bearing input for precise estimates is methodologically weak. The revised estimate treats them as directional, not precise.

What remains undisputed

The Klarna engineer's three GitHub issues are real primary sources. The breakage they describe is real. The AdGuard precedent for PayMaya (paymaya.customlinks.appsflyer.com in exceptions.txt) is verified. The problem exists — the question is purely about blast radius, which is narrower than initially estimated.

Revised bottom line: The blocking is real but narrow. It affects users of specific DNS services with server-side CNAME-cloaking detection (NextDNS configured profiles, possibly AdGuard DNS server-side). The original estimate of 2.6–10% was too high. A more honest range is 0.2–1%, acknowledging uncertainty about how many NextDNS/AdGuard configurations actually trigger the block.

💡 Bottom Line

l.klarna.com is blocked for an estimated 1–3% of Klarna's 119M users (~1.2–3.5M people), primarily through NextDNS and AdGuard DNS which both have confirmed CNAME-based blocking of the AppsFlyer chain. The original estimate of 2.6–10% was too high because it counted Pi-hole users (not affected — no CNAME following) and browser extensions (not affected — no appsflyer in EasyPrivacy/Disconnect).


The problem is real but narrow: it affects users of DNS blockers with CNAME uncloaking technology (NextDNS, AdGuard DNS, ControlD, AdGuard Home). The fix is straightforward — Klarna needs either a server-side proxy (A record instead of CNAME) or per-alias exceptions like AdGuard already grants for PayMaya.
